# Phase 32-01 PLAN — Backend Codex OAuth lifecycle and upstream-auth errors

## Objetivo

Criar a base backend definitiva para o channel 5 `OpenAI - Codex`: metadata sanitizada, refresh com erro claro, regeneration OAuth start/complete, probe controlado e erro upstream-auth especifico para `token_invalidated`/`refresh_token_invalidated`.

## Escopo

- `router/channel-router.go`
- `controller/codex_oauth.go`
- `controller/channel.go`
- `controller/codex_usage.go`
- `service/codex_oauth.go`
- `service/codex_credential_refresh.go`
- `service/codex_credential_refresh_task.go`
- `service/codex_catalog.go`
- `service/error.go`
- `types/error.go`
- `relay/responses_handler.go`
- `relay/channel/codex/`

## Tarefas

1. Registrar rotas admin Codex:
   - `GET /api/channel/:id/codex/credential`
   - `POST /api/channel/:id/codex/refresh`
   - `POST /api/channel/:id/codex/regenerate/start`
   - `POST /api/channel/:id/codex/regenerate/complete`
   - `POST /api/channel/:id/codex/probe`
2. Reaproveitar `StartCodexOAuthForChannel` e `CompleteCodexOAuthForChannel` para regeneration, ajustando nomes/respostas se necessario.
3. Implementar DTO sanitizado de credential metadata sem `access_token`, `refresh_token`, `id_token` ou headers.
4. Criar erro tipado em `service/codex_oauth.go` para respostas OAuth upstream nao-2xx, preservando `status`, `error`, `error_description` e preview sanitizado.
5. Implementar classificador `requires_regeneration` para:
   - refresh_token ausente
   - refresh_token invalidado
   - token_invalidated
   - 401/403 upstream Codex
6. Persistir health nao secreto em settings do channel sem migration e sem sobrescrever settings existentes.
7. Normalizar erro relay Codex para codigo distinto, por exemplo `codex_upstream_token_invalidated`, mantendo HTTP 401/403 upstream e mensagem operacional.
8. Atualizar `controller/codex_usage.go` e `service/codex_catalog.go` para registrar/retornar refresh failure classificado quando o refresh falhar.
9. Cobrir testes Go focados.

## Validacao da Wave

Comandos obrigatorios:

```bash
git diff --check -- router/channel-router.go controller/codex_oauth.go controller/channel.go controller/codex_usage.go service/codex_oauth.go service/codex_credential_refresh.go service/codex_credential_refresh_task.go service/codex_catalog.go service/error.go types/error.go relay/responses_handler.go relay/channel/codex
./scripts/podman-admin.sh profile-run -- bash -lc 'GOCACHE=$(mktemp -d) go test ./controller ./service ./relay ./relay/channel/codex -count=1'
```

Validacoes logicas:

- Refresh sem `refresh_token` retorna mensagem acao: regenerar.
- `refresh_token_invalidated` retorna codigo especifico e nao mascara como erro generico.
- Relay Codex com upstream `token_invalidated` retorna codigo upstream-auth, nao erro de API key interna do Router.
- Metadata endpoint nunca retorna tokens.

## Criterio de Conclusao

Backend pronto para a UI consumir refresh/regenerate/probe com mensagens operacionais claras e sem expor segredo.
