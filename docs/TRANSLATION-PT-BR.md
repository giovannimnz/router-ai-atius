The pt file is **larger than the source** because Brazilian Portuguese sometimes needs more words for the same concept (e.g. compound noun phrases that English packs into one word).

## How the translation was made (no paid service)

1. **Source**: `web/default/src/i18n/locales/en.json` is the single source of truth. Every UI string the app shows is keyed by its English text.
2. **Schema**: flat JSON under a `translation` namespace. Keys are **English source strings** (the same text shown in English), values are the pt translation.
3. **Process**:
   - A native Brazilian Portuguese speaker (the project maintainer) translated each key.
   - Cross-referenced with the upstream QuantumNous/new-api pt-BR translation to stay consistent with the open-source community's vocabulary.
   - Reviewed for consistency: same English term always translates the same way (e.g. "channel" → "canal", never "via" or "meio").
4. **Tech used**: zero. No DeepL, no Google Translate API, no LLM translation. Hand-curated with feedback from real users (the Atius mesh in Brazil).
5. **Verification**:
   - `bun run i18n:sync` checks that the pt file is in sync with en (no missing keys, no orphan keys).
   - Manual UI walkthrough on the dev server to confirm idiomatic phrasing.

### Things we kept on purpose

- **Technical acronyms in English** (API, HTTP, ID, URL, JSON). Translating them creates friction for developers reading docs.
- **Brand names** (Atius, MiniMax, DeepSeek). Don't translate.
- **False friends** in mind: "expoente" (exponent, not "exposition"), "salvar" (save, not "salve" which is an interjection).

### Things we changed on purpose

- **"Del" → "Excluir"** (the upstream uses "Del" sometimes, which is an anglicism).
- **"Add" → "Adicionar"** (kept full word, not "Add" in caps).
- **"OK" / "Cancel"** buttons are kept in English because that's the universal pattern users expect.

## How to add or fix a translation

1. Open `web/default/src/i18n/locales/en.json` and find the English key.
2. Edit `web/default/src/i18n/locales/pt.json` to add or change the value.
3. Run `bun run i18n:sync` from `web/default/` to verify no keys are missing.
4. Test in the dev server (`bun run dev`) — the language switcher in the top-right should show "Português" when the user picks BR.
5. Commit with the change.

## How the language switcher works

The user's preferred language is stored in their user settings in the new-api Go DB. When the user picks "Português" in the language switcher, the choice is sent to the API and the next page load uses `pt.json`.

For unauthenticated users (login page), the choice is in `localStorage` only — they have to re-pick it on each device.

The `normalizeInterfaceLanguage()` helper in `web/default/src/i18n/languages.ts` handles the edge cases:
- `pt-BR` (mixed case from i18next storage) → `pt-br` (lowercase)
- `pt` (region-less, from some browser locale detection) → falls back to `pt-br` via i18next's built-in fallback chain
