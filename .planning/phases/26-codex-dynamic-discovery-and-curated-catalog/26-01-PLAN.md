<objective>
Implement a dynamic Codex discovery pipeline that feeds the local curated catalog safely.

The route `/v1/models` must remain local and deterministic, but the underlying Codex model set should become dynamically discoverable, enriched with a second metadata source plus local overrides, and auto-promoted only after successful validation.
</objective>

## Workstream 1 — Codex discovery source

Create a Source A abstraction for Codex model availability.

Deliverables:
- dynamic discovery service for channel `5` / type `57`
- raw candidate ids + discovery metadata
- replacement path for static-only `fetchCodexModelIDs()`

Acceptance:
- discovery returns account-aware candidate models from the active Codex credential path
- no request-time dependency is added to `/v1/models`
- dynamic discovery is the primary availability source; the local cache acts as fallback when discovery is unavailable

## Workstream 2 — Local storage and state model

Create the local cache/source-of-truth shape for this pipeline.

Deliverables:
- persisted raw discovery snapshot
- persisted merged candidate state
- local override/default-policy store
- explicit states: discovered / enriched / validated / promoted / rejected

Acceptance:
- the system can tell “unchanged”, “changed but unvalidated”, and “validated/promotable”

## Workstream 3 — Source B enrichment and merge

Use an additional metadata source and join it with discovery.

Deliverables:
- adapter for existing upstream metadata sync surface where possible
- local Codex override layer
- merge precedence rules

Must enrich:
- a parent `context_window` field/group
- `max_tokens`
- `max_completion_tokens`

Acceptance:
- merged Codex candidates always have stable endpoint preference and enough metadata to decide promotion

## Workstream 4 — Probe gate

Validate candidate availability before publication.

Deliverables:
- minimal live request that asks the candidate to return only `Ok`
- classification of validation outcomes
- promotion blocked on success

Acceptance:
- failed candidates remain cached but unpublished
- successful candidates may be promoted automatically

## Workstream 5 — Daily scheduler

Run exactly one daily job at `04:00`.

Responsibilities:
- discover
- enrich
- diff
- validate
- promote
- refresh caches
- notify only on meaningful change

Acceptance:
- no rewrite when upstream state is unchanged
- no second generic scheduler for the same concern

## Workstream 6 — Catalog/public integration

Wire promoted Codex entries into the existing curated catalog.

Deliverables:
- Codex text models prefer `openai-response`
- existing ordering/pricing/provider grouping preserved
- default model remains a local policy

Acceptance:
- `/v1/models` only shows promoted/validated Codex models
- Codex endpoint metadata remains curated and deterministic

## Verification

### Code-level

- service tests for dynamic discovery normalization
- merge precedence tests
- validation gate tests
- scheduler “write only on change” tests
- endpoint preference tests for Codex
- metadata enrichment tests for `context_window`, `max_tokens`, `max_completion_tokens`
- tests that `context_window` is emitted as the parent field/group for the token-window metadata block

### Runtime-level

- simulate no-change run -> no catalog rewrite
- simulate new valid model -> candidate promoted
- simulate invalid candidate -> cached, not promoted
- confirm `/v1/models` stays local and contract-compatible

## Explicit non-goals

- official OpenAI docs / CI / release workflow rewrite
- public preview surfacing of unvalidated models
- request-time upstream discovery
