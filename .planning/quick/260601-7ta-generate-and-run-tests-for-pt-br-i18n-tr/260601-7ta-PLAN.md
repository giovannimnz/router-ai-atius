# PLAN: Generate and Run Tests for pt-BR i18n Translation

Task code: 260601-7ta
Scope: web/default (Bun + Rsbuild + React 19 + i18next)
Goal: Automated test suite that proves the pt-BR translation is fully in sync (0/0/0), all 7 locales load, the language switcher wires the right codes, the hero translates, and `localStorage` persistence works. All tests must pass on the current codebase.

Baseline facts gathered from repo:
  - 7 locales: en, zh, fr, ja, pt-BR, ru, vi (`web/default/src/i18n/locales/`).
  - pt-BR is the auto-detected base locale (4525 keys under `translation.*`).
  - Current sync report (`src/i18n/locales/_reports/_sync-report.json`):
      en/fr/ja/ru/vi/zh all show `missingCount: 1055` (because pt-BR is the base).
      pt-BR itself shows `0/0/0` — that is the success criterion.
  - `web/default/src/i18n/config.ts` exports `resources` keyed by `en | zh | fr | ja | 'pt-BR' | ru | vi`.
  - `web/default/src/i18n/languages.ts` exports `INTERFACE_LANGUAGE_OPTIONS` with the 7 codes.
  - `web/default/src/components/language-switcher.tsx` ships its own local `languages` array including `'pt-BR'`.
  - `web/default/src/features/home/components/sections/hero.tsx` is the canonical i18n consumer (uses `useTranslation` and `t()`).
  - No vitest is currently installed; one test exists (`dropdown-menu.test.tsx`) using `node:test`. AGENTS.md lists vitest as optional. We will install vitest + @testing-library/react + jsdom for this task.

Test framework decision: vitest. The task brief specifies vitest; AGENTS.md §3.14 endorses it; the existing `dropdown-menu.test.tsx` test will be ported to vitest as part of Task 1 so the suite is uniform.

TASK 2 — Add unit tests for locale JSON integrity and config wiring
--------------------------------------------------------------------
Files:
  - web/default/src/i18n/__tests__/locales-integrity.test.ts (new)
  - web/default/src/i18n/__tests__/languages-config.test.ts (new)
  - web/default/src/components/__tests__/language-switcher.test.tsx (new)

Action — `locales-integrity.test.ts` (covers Tests 1-4 of the brief):
  1. Load every `*.json` under `src/i18n/locales/` (exclude `_extras` and `_reports`) and assert each `JSON.parse`s without error and is a plain object. (Test 1)
  2. For each non-`en` locale, recursively walk both trees and assert:
       - same set of leaf paths,
       - identical leaf count.
     Report the first divergence with the path. (Test 2)
  3. Assert `pt-BR.json` has 0 missing keys relative to the auto-detected base (= itself), and 0 extras, by re-implementing the same diff logic the sync script uses (or by importing the sync script's pure helpers). (Test 3)
  4. Run an "untranslated" scan over `pt-BR`:
       - For every leaf whose en value is a non-brand string (length >= 6, not URL/email/placeholder, not in `BRAND_AND_LITERAL_KEYS`), assert the pt-BR value is NOT byte-equal to the en value.
       - Collect (don't fail) the count, but assert count == 0 for `pt-BR`. (Test 4)

Action — `languages-config.test.ts` (covers Tests 5-7):
  5. Import `INTERFACE_LANGUAGE_OPTIONS` and assert it contains exactly the 7 expected codes: `en, zh, fr, ja, 'pt-BR', ru, vi` (order-independent). (Test 5)
  6. Read `src/components/language-switcher.tsx` as a string and assert it contains the literal string `"pt-BR"` and a code entry `code: 'pt-BR'`. (Test 6)
  7. Read `src/i18n/config.ts` as a string and assert it contains `'pt-BR': ptBR` and that `supportedLngs` includes `'pt-BR'`. Also import the `resources` object at runtime and assert `resources['pt-BR']` is defined and has the same key set as `resources.en`. (Test 7)

Action — `language-switcher.test.tsx` (covers Test 6 interaction):
  - Render `<LanguageSwitcher />` with `useAuthStore` mocked to logged-out.
  - Open the dropdown and assert the rendered menu includes the text "Português" (the `pt-BR` label from `INTERFACE_LANGUAGE_OPTIONS`).
  - Click the item and assert `i18n.changeLanguage` was called with `'pt-BR'`.
  - Mock `api.put` and assert it was NOT called (user is null).

Verify:
  - `bun run test` — all new tests pass.
  - Snapshot the `pt-BR` untranslated count to confirm it stays 0.

Done when:
  - Tests 1-7 from the brief are covered by passing tests.

TASK 3 — Add integration tests: i18n runtime behavior (Tests 8-9)
--------------------------------------------------------------------
Files:
  - web/default/src/i18n/__tests__/i18n-runtime.test.tsx (new)
  - web/default/src/i18n/__tests__/hero-render.test.tsx (new)

Action — `i18n-runtime.test.tsx` (Test 9 — persistence / changeLanguage):
  1. Use the `loadI18n('en')` helper from Task 1 to build a real i18next instance backed by the actual JSON files.
  2. Assert `i18n.t('Docs')` resolves to the English value (sanity).
  3. Call `await i18n.changeLanguage('pt-BR')`; assert:
       - `i18n.language === 'pt-BR'`
       - `i18n.t('Docs')` returns a Portuguese value (assert it is NOT byte-equal to the English value and matches a known pt-BR string from `pt-BR.json`).
  4. Assert the i18n instance stores the chosen language in `localStorage` under the key used by `i18next-browser-languagedetector` (default `i18nextLng`). This satisfies the "localStorage persistence" requirement in spirit; we instantiate LanguageDetector in the helper.

Action — `hero-render.test.tsx` (Test 8 — Hero translates under pt-BR):
  1. Build a wrapper that initializes i18next with `lng: 'pt-BR'` and provides it via `I18nextProvider`.
  2. Mock `@/hooks/use-status` so `status` is `undefined` (default `docs_link`).
  3. Render `<Hero />` with `isAuthenticated={false}` and assert the document contains at least one Portuguese-only substring that appears in `pt-BR.json` and NOT in `en.json` (e.g. pick a stable distinguishing key from the hero strings — verify by grep beforehand; for example the translated value for `"Go to Dashboard"` or `"Get Started"`).
  4. Snapshot/assert that none of the untranslated `t('...')` calls in hero.tsx render the English fallback when `lng === 'pt-BR'`.

Verify:
  - `bun run test` — all integration tests pass.
  - `bun run typecheck` still passes (strict TS, no `any` in test code).

Done when:
  - Tests 8 and 9 are green and the Hero actually renders Portuguese under `lng: 'pt-BR'`.
